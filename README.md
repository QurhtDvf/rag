# RAG（Retrieval-Augmented Generation）の歴史

## 1. RAGとは何か

**RAG（Retrieval-Augmented Generation）** とは、LLMの推論時に外部の知識ベースから関連情報を動的に検索し、その内容をプロンプトに組み込んで回答を生成する技術パラダイムである。

### LLMの3つの根本的な限界

| 課題 | 内容 | RAGによる解決 |
|------|------|--------------|
| 知識の陳腐化 | 学習データのカットオフ以降の情報を知らない | 外部DBをリアルタイムに検索して補完 |
| ハルシネーション | 存在しない事実をもっともらしく生成する | 検索結果という「証拠」に基づいて生成 |
| ブラックボックス性 | どこから情報を得たか追跡できない | 参照元ドキュメントを明示可能 |

### RAGの基本パイプライン

```
ユーザークエリ
    ↓
[Retriever] クエリをベクトル化 → 外部知識ベースを検索 → 関連チャンクを取得
    ↓
[Generator] クエリ + 取得チャンク → LLMに入力 → 回答生成
    ↓
出力（参照元付き）
```

RAGの核心的なアイデアは「**パラメトリックメモリ**（モデルの重みに蓄積された知識）と**ノンパラメトリックメモリ**（外部検索可能なドキュメント群）を組み合わせる」点にある。

## 2. RAG以前：情報検索（IR）の歴史

RAGはある日突然生まれた技術ではなく、数十年にわたる情報検索（IR）研究の集大成である。

### 情報検索技術の変遷

| 時代 | 技術 | 概要 |
|------|------|------|
| 1950s–60s | TF-IDF / ベクトル空間モデル | Hans Peter LuhnやGerald Saltonが基礎を確立。単語の出現頻度で文書を比較 |
| 1970s–80s | ブール検索 / 転置インデックス | キーワードの AND/OR/NOT 演算で検索。現在の全文検索の基礎 |
| 1990s | BM25 | TF-IDFを改良したスパース検索手法。現在もElasticsearchなどで広く使われる |
| 2000s | 学習ランキング（LTR） | 機械学習で検索結果の順位を最適化 |
| 2017– | 密ベクトル検索（Dense Retrieval） | Transformerを用いて文書をベクトル化し意味的類似性で検索 |

### RAGの直接の先祖：Open-Domain QA

2019年前後に、「ウィキペディア全体を参照して質問に答える」Open-Domain QAの研究が盛んになった。

- **DPR（Dense Passage Retrieval, Karpukhin et al. 2020）:** 質問と文書を双方向にエンコードし、内積で類似度を計算する密検索手法を確立
- **REALM（Guu et al. 2020）:** 検索を事前学習に組み込んだ先駆的モデル
- **ORQA（Lee et al. 2019）:** 教師なしで検索器を学習する手法

これらの研究が、Lewis et al. のRAGの直接の基盤となった。

## 3. RAGの誕生（2020）

### 論文："Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks"

2020年5月、Meta AI（旧Facebook AI Research）のPatrick Lewisらが発表した論文（NeurIPS 2020）が、RAGという概念を正式に定義した。

```
論文: "Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks"
著者: Patrick Lewis et al. (Meta AI / UCL / NYU)
発表: NeurIPS 2020
arXiv: 2005.11401
```

### 論文の核心

Lewisらは2種類のRAGモデルを提案した。

- **RAG-Sequence:** 取得した文書全体をもとに回答全体を生成
- **RAG-Token:** 各トークンの生成時に異なる文書を参照できる

オープンドメインQAの3つのベンチマーク（Natural Questions、WebQuestions、CuratedTrec）で当時の最高性能を達成し、LLMの「閉じた本」的な生成を「開かれた参照付き」生成へと転換する道を示した。

### 命名の裏話

Lewisは後にインタビューで「RAGという略称があれほど広まるとは思っていなかった。もっとよい名前を考えるつもりだったが、論文を書く段階では誰も良いアイデアを持っていなかった」と語っている。

## 4. RAGアーキテクチャの進化

RAGの研究パラダイムは、**Naive RAG → Advanced RAG → Modular RAG** という3段階で進化してきた。

### 4.1 Naive RAG（2020–2022）

ChatGPTの台頭とともに急速に普及した最初期の実装形式。構造はシンプルな3ステップで構成される。

```
[Indexing]   ドキュメント → チャンク分割 → ベクトル化 → ベクトルDBに格納
[Retrieval]  クエリをベクトル化 → コサイン類似度で上位K件を取得
[Generation] クエリ + 取得チャンク → LLMに入力 → 回答
```

**限界:**
- 検索精度の低さ（意味的にズレたチャンクを取得しやすい）
- チャンクの断片化（重要情報が複数チャンクに分散する）
- ハルシネーション（取得チャンクと矛盾する回答を生成する場合がある）

### 4.2 Advanced RAG（2022–2023）

Naive RAGの弱点を補う前処理・後処理を追加した改良版。

**検索前（Pre-Retrieval）の最適化:**
- **Query Rewriting:** クエリを言い換えて検索精度を向上
- **HyDE（Hypothetical Document Embeddings）:** LLMに仮想回答を生成させ、その回答で検索する
- **Query Decomposition:** 複雑な質問を複数のサブクエリに分解

**検索後（Post-Retrieval）の最適化:**
- **Re-ranking:** 取得したチャンクをクロスエンコーダで再スコアリング
- **Context Compression:** 不要な文を削除し、関連部分だけに圧縮
- **RAG-Fusion:** 複数クエリの検索結果をRRF（Reciprocal Rank Fusion）でマージ

### 4.3 Modular RAG（2023–）

RAGを固定パイプラインではなく「交換可能なモジュールの組み合わせ」として設計するアーキテクチャ（Gao et al. 2024）。

```
[Search Module]   ウェブ検索・DBクエリ・APIコールなど多様な検索手段
[Memory Module]   短期・長期メモリの管理
[Fusion Module]   複数検索結果の統合
[Routing Module]  クエリ難易度に応じてパイプラインを動的に選択
[Predict Module]  LLMが自ら検索の必要性を判断
```

### 4.4 特殊化したRAGの系譜

| 手法 | 概要 |
|------|------|
| **GraphRAG** (Microsoft, 2024) | 知識をグラフ構造で管理し、エンティティ間の関係を活用 |
| **Self-RAG** (Asai et al. 2023) | LLM自身が「検索すべきか」「回答は正確か」を自己評価 |
| **Adaptive RAG** (Jeong et al. 2024) | クエリの難易度を分類し、単純な質問には検索不要と判断 |
| **LongRAG** (Jiang et al. 2024) | より長い検索単位（Long Retrieval Unit）を導入 |
| **Agentic RAG** (2024–) | ReActフレームワークと組み合わせ、多段階の検索・推論を実行 |

## 5. RAGエコシステムの拡大（2022–2023）

### フレームワークの整備

ChatGPT（2022年11月）の登場によりLLMアプリ開発が急拡大し、RAGを実装するためのフレームワークが相次いで登場した。

| フレームワーク | 登場 | 特徴 |
|--------------|------|------|
| LlamaIndex | 2022年末 | ドキュメントのインデックス化と検索に特化 |
| LangChain | 2022年末 | LLMアプリの全パイプラインをチェーンとして構築 |
| Haystack | 2020年〜 | 本格的な検索パイプラインの構築を支援 |

### ベクトルDBの台頭

RAGの普及に伴い、大規模な密ベクトルを高速に検索するベクトルデータベースが産業として勃興した。

- **Pinecone**（2021年〜）: マネージド型ベクトルDB
- **Weaviate / Qdrant / Milvus**: オープンソース系ベクトルDB
- **pgvector**: PostgreSQL拡張としてのベクトル検索

## 6. 「RAG is Dead」論争（2024–2025）

### 論争の発端

「RAG is Dead」という主張は、コンテキストウィンドウが大幅に拡大するたびに繰り返し浮上してきた議論である。

| モデル | コンテキスト長 | 登場 |
|--------|------------|------|
| GPT-3 | 4K tokens | 2020 |
| GPT-4 Turbo | 128K tokens | 2023 |
| Gemini 1.5 Pro | 1M tokens | 2024 |
| Gemini 2.0 Flash | 2M tokens | 2025 |
| Llama 4 Scout | 10M tokens | 2025 |

「1000万トークンのコンテキストウィンドウがあれば、知識ベース全体をプロンプトに詰め込める。RAGの複雑なパイプラインは不要になるのでは？」という主張が、議論の核心である。

### Long Context派の主な論拠

- ベクトルDB・チャンク分割・再ランキングなどの複雑な実装が不要になる
- チャンク分割による情報断片化の問題を根本的に回避できる
- Li et al. (2024) の研究では「リソースが十分な場合、Long Contextの方がRAGより性能が高い」という結果も示された

### 「RAG is Dead」に反論する論拠

**1. コストと速度の問題**

長いコンテキストの処理コストは**非線形に増大**し、経済的に非現実的なケースが多い。2025年初頭時点で、Gemini 2.0 Flashを36万トークンで使用すると30秒以上のレイテンシが報告されている。一方、最適化されたRAGシステムでは検索ステップ300〜600ms・生成込みで約1秒の応答が実現できる。

**2. "Lost in the Middle" 問題**

Stanfordの研究が示すように、LLMは長いコンテキストの中間部分にある情報を十分に活用できない。コンテキストの最初と最後への偏重（プライマシー・リセンシーバイアス）により、100万トークン全体を詰め込んでも精度が大幅に劣化するU字型のパフォーマンス曲線が観測されている。

**3. データスケールの問題**

実世界の企業の知識ベースはメガバイトからテラバイト規模に及ぶ。どれだけコンテキストが拡張されても、データの増加速度はコンテキスト長の拡大を常に上回ってきた歴史がある。

**4. 動的データへの対応**

頻繁に更新されるデータソースに対して、都度すべてをコンテキストに詰め込む方式はキャッシュの有効期限問題を引き起こす。RAGなら知識ベースのインデックスを更新するだけでよい。

### 研究者コミュニティの見解

2024〜2025年にかけて複数の比較研究が行われ、「一方が常に優れている」わけではないことが明らかになっている。

> *"LC generally outperforms RAG in question-answering benchmarks, especially for Wikipedia-based questions. However, RAG has advantages in dialogue-based and general question queries."*
> — Li et al. (2024), arXiv:2501.01880

> *"There was no one-size-fits-all solution. The choice ultimately depends on the model size, task type, and context length."*
> — LaRA benchmark study (Li et al., 2025)

### 現在のコンセンサス

**「Naive RAGは死んだ。Sophisticated RAGは進化している。」**

2026年時点での実務的な判断基準はおよそ以下の通りである。

| 条件 | 推奨アプローチ |
|------|--------------|
| 静的ドキュメント、100件以下、総量100K tokens以内 | Long Context で十分 |
| 1,000件以上のドキュメント、動的データ | RAG が優位 |
| レイテンシが重要（1秒以下の応答） | RAG が優位 |
| コスト効率が最重要 | RAG が優位（最大1,250倍のコスト差とも） |
| プロトタイプ・一回限りの分析 | Long Context の方がシンプル |

## 7. Long Context vs. RAG：技術的比較

### アーキテクチャ比較

```
Long Context アプローチ:
  全ドキュメント → トークン化 → コンテキストウィンドウに詰め込む → LLM推論
  ※ シンプルだが、大規模データには非現実的

RAG アプローチ:
  全ドキュメント → インデックス化（事前）
  クエリ → 検索 → 関連チャンクのみ取得 → LLM推論
  ※ 複雑だが、任意規模のデータに対応可能
```

### コスト・精度・速度の比較

| 指標 | Long Context | RAG |
|------|-------------|-----|
| セットアップコスト | 低い（インフラ不要） | 高い（インデックス・ベクトルDB必要） |
| クエリコスト | 高い（全トークン処理） | 低い（関連チャンクのみ処理） |
| レイテンシ | 高い（非線形に増大） | 低い（300ms〜1秒程度） |
| 動的データ対応 | 困難 | 容易（インデックス更新のみ） |
| 最大データ規模 | コンテキスト長に依存 | 事実上無制限 |
| 情報精度 | Lost in the Middleリスクあり | 関連情報のみ絞り込めるため高精度 |

## 8. RAGの現在地と今後（2025–）

### 「Context Engine」への進化

2025〜2026年にかけて、RAGは単体の検索ツールから「**Context Engine**（文脈組み立て基盤）」へと概念が拡張されつつある。単に文書を検索するのではなく、AIが必要なタイミングに必要な情報を最適な形で組み立てる基盤インフラとして再定義されている。

### Agentic RAGの台頭

LLMエージェントの普及に伴い、RAGは複数ステップの推論と検索を組み合わせた「Agentic RAG」へと発展している。

```
クエリ受信
  → LLMが検索戦略を計画
  → 複数の検索を実行（ウェブ、DB、APIなど）
  → 中間結果を推論
  → 追加検索の必要性を自己判断
  → 最終回答の生成
```

### 採用状況

2024年以降、RAGフレームワークの採用は急速に拡大しており、本番運用のLLMアプリケーションの60%がRAGを採用しているとの報告もある。特に中大規模企業での知識集約型ワークフローにおいて30〜70%の効率改善が報告されている。

### 今後の研究方向

- **マルチモーダルRAG:** テキストに加え、画像・動画・音声も検索対象に
- **Streaming RAG:** リアルタイムデータストリームへの対応
- **Graph RAG の成熟:** 知識グラフを活用した関係推論の強化
- **Self-improving RAG:** フィードバックループによって検索精度を自律的に改善
- **Long Context × RAG のハイブリッド:** 両者を組み合わせ、短い文書にはLong Context、大規模DBにはRAGを使い分ける Self-ROUTE などの手法

## 9. まとめ

### RAGの歴史的変遷

| 時期 | 技術・出来事 | 意義 |
|------|------------|------|
| 1950s–90s | TF-IDF、BM25、転置インデックス | 情報検索の基礎が確立 |
| 2019–20 | DPR、REALM、ORQA | 密ベクトル検索とニューラル統合の実現 |
| 2020 | Lewis et al. の論文（NeurIPS） | RAGの概念が正式に確立 |
| 2022 | ChatGPT登場、LangChain/LlamaIndex | RAGの実用化・民主化が加速 |
| 2022–23 | Advanced RAG（再ランキング、HyDE等） | Naive RAGの限界を克服 |
| 2023–24 | Modular RAG、GraphRAG、Self-RAG | アーキテクチャの高度化・特殊化 |
| 2024 | 「RAG is Dead」論争の本格化 | Long Context vs. RAGの比較研究が増加 |
| 2025– | Agentic RAG、Context Engine | RAGのエージェント化・インフラ化 |

### 結論

「RAG is Dead」という主張は、**Naive RAGの限界を正しく指摘している**一方で、RAG技術全体の終焉を意味しない。コンテキストウィンドウの拡大はRAGが不要になる文脈を確かに生み出したが、大規模・動的・コスト重視のエンタープライズ用途ではRAGの優位性は依然として確固たるものである。

技術の本質的な問いは「RAGかLong Contextか」という二項対立ではなく、**「どのユースケースにおいて、どの手法が最もコスト・精度・レイテンシのバランスを満たすか」** という実務的な最適化問題に移行している。
